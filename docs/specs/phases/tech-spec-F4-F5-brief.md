# Фазы 4–5: переработанная постановка от заказчика

## Назначение документа
Кратко фиксирует новые вводные по Фазе 4 и Фазе 5 в структурированном виде и содержит перечень нетехнических вопросов для уточнения перед подготовкой полноценного ТЗ.

## Фаза 4. Публичный контур контента и Медиацентр

1. Создать новую главную страницу платформы как основную точку входа для всех ролей.
2. Запустить отдельный раздел `Медиацентр` с подразделами:
   - новости о прошедших и предстоящих мероприятиях;
   - блок обучения от вузов Кластера;
   - публикации мероприятий.
3. Для блока обучения от вузов Кластера отображать предложения от вузов-партнеров по профильным направлениям.
4. Для каждого предложения по обучению обязательно показывать даты начала и окончания курса.
5. Предусмотреть возможность публикации мероприятий ограниченной категорией участников (на первом этапе: представители вузов, включая ректоров и деканов).
6. Права на публикацию мероприятий предоставлять только после ручного подтверждения администратором.
7. Предусмотреть бизнес-механизм запроса права публикации (вариант: отдельная заявка или подтверждение при регистрации).
8. Добавить модерацию материалов от стейкхолдеров:
   - для `запросов`;
   - для `мероприятий`.
9. Для модерации использовать единый цикл:
   - `Черновик`;
   - `На модерации`;
   - `Опубликовано`;
   - `Возвращено на доработку`;
   - `Отклонено`.
10. По каждому решению модерации фиксировать причину решения и информировать автора материала.

## Фаза 5. Сервис сопровождения и База знаний

1. Создать контур сопровождения проектов по ролям:
   - трекер;
   - ментор;
   - эксперт.
2. Зафиксировать регламент участия для каждой роли:
   - зона ответственности;
   - формат участия;
   - ожидаемый результат;
   - точки контроля.
3. Проработать улучшения совместной работы команд и участников внутри проекта.
4. Развить сервис информирования о статусах реализации и этапах внутреннего контроля.
5. Создать раздел `База знаний` с отраслевыми и методическими материалами по направлениям деятельности Кластера.
6. База знаний наполняется участниками, но публикации проходят обязательную премодерацию.

## Очень краткое ТЗ главной страницы (Фаза 4)

1. Назначение: единая входная страница для новых и действующих пользователей.
2. Блоки страницы (сверху вниз):
   - рекламный баннер/слайдер (анонсы мероприятий или обучения);
   - короткий блок «О платформе»;
   - блок «Важные запросы сейчас» (3 карточки);
   - блок «Новые проекты» (3 карточки из одобренных идей, доступных для участия);
   - подвал.
3. Источник карточек:
   - запросы — по приоритету/актуальности;
   - проекты — последние одобренные идеи.
4. Действия:
   - переход в детали запроса/проекта;
   - переход из баннера в страницу мероприятия или курса.
5. Критерии приемки:
   - все блоки отображаются в указанном порядке;
   - в каждом блоке карточек выводится по 3 элемента;
   - все CTA-кнопки и карточки кликабельны;
   - страница корректно отображается на desktop и mobile.

## Уточняющие вопросы к заказчику (нетехнические)

### 1) По новой главной странице и Медиацентру
1. Какой главный приоритет главной страницы на запуске: привлечение идей, мероприятия, обучение или кейсы?
2. Кто является владельцем контента в Медиацентре: единая редакция или разные владельцы по подразделам?
3. Нужно ли включать в Медиацентр только анонсы или также отчеты/итоги прошедших активностей?

### 2) По публикации мероприятий внешними участниками
1. Какие категории пользователей, кроме представителей вузов, потенциально должны получить право публикации в будущем?
2. Право публикации выдается бессрочно или на ограниченный период?
3. Какой минимальный набор подтверждающих данных обязателен для получения права публикации?

### 3) По модерации материалов стейкхолдеров
1. Какие изменения в запросах и мероприятиях требуют обязательной модерации, а какие можно публиковать сразу?
2. Кто принимает финальное решение по модерации (роль/подразделение)?
3. Нужен ли единый SLA по сроку модерации или разные SLA для запросов и мероприятий?
4. Сколько итераций доработки допустимо до финального отклонения материала?
5. Нужно ли вести отдельный аудит-реестр решений по модерации для руководства?

### 4) По Фазе 5 (сопровождение и база знаний)
1. Какие обязательные функции должны выполнять трекер, ментор и эксперт на первом релизе?
2. Для каких категорий проектов сопровождение обязательно, а для каких опционально?
3. Кто несет ответственность за финальное качество материалов в Базе знаний?
4. Какие материалы в Базе знаний приоритетны на запуске (методики, кейсы, шаблоны, учебные материалы)?

- Заказчик дал ответы на уточняющие вопросы по приоритетам, ролям и правилам модерации.
- На основе подтвержденного брифа можно готовить полноценные ТЗ `tech-spec-F4.md` и `tech-spec-F5.md`.

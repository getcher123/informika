# Работа с удалёнными разработчиками через Git

## Ветвление и доступ
- `main` защищён: изменения — только через PR и обязательное ревью от Lead.
- Удалённые разработчики работают через fork.
- Поток: создают ветку `feat/<task>` или `fix/<task>` → пушат код → открывают PR.
- CI запускает проверки, Lead проводит ревью → merge в `main`.

## CI/CD
- CI: линтеры (`php-cs-fixer`/`eslint`/`stylelint`), unit/интеграционные тесты (если есть), сборка/проверка артефактов.
- CD (полуавтоматический по тегу):
  - `staging` — автодеплой из `main` (по merge).
  - `prod` — вручную по тегу/релизу (Lead).
- Секреты — через переменные окружения в CI/CD.
- `.env` и доступы — вне Git, выдаёт Lead.

## Постановка задач для ad‑hoc Dev (удалёнщик)
- Карточка содержит: контекст, ожидаемый результат (AC), ограничения, макап/скрин, критерии приемки (DoD).
- Размер задачи ≤ 1–2 дня. Если больше — декомпозировать.
- Согласование времени/ставки — у PM (см. юр. блок).

## Шаблон PR‑чек‑листа
- Воспроизводимость описана.
- Скриншоты/видео результата приложены.
- Не ломает существующие сценарии.
- Тест/проверка на `staging` пройдена.
- Обновлён CHANGELOG (если нужно).

## Коммуникации и ритуалы
- Командный чат (Telegram/Slack) + короткая daily async‑форма: «Что сделал/делаю/риски/блокеры».
- Еженедельное планирование (30–45 мин): PM + Lead.
- Демо раз в неделю (15–30 мин): показать, что уходит в `staging`/`prod`.
- Канбан‑доска в GitHub/Trello: одна колонка = один статус (см. статусы задач).

## Мини‑регламенты
- Срок ревью PR: до 1 рабочего дня.
- Лимиты WIP: не более 2 активных задач на Dev.
- Размер PR: ≤ 300 строк diff.
- Доступы: только по роли; доступ к прод — у Lead.
- Бэкапы/откат: чек‑лист релиза с планом rollback.
- Онбординг/оффбординг: чек‑лист доступов, ключей, секретов; передача задач фиксируется в тикетах.

# Краткое техническое задание

## 1. Назначение и цели
Платформа предназначена для управления инновационным циклом в организации: от запроса на инновацию (инициатива стейкхолдера) → сбора идей участников → превращения подтвержденной идеи в проект и командной работы над ним в Bitrix. Публичные защиты на НТС находятся вне рамок данной платформы.

## 2. Роли и доступы
- Стейкхолдер — формирует запросы на инновации, подтверждает идеи, принимает ключевые решения по проектам.
- Участник — подаёт идеи на конкретные запросы, участвует в проектах.
- Модератор — обеспечивает корректность публикаций, следит за соблюдением правил.
- Эксперт (эдвайзер) — консультирует; подключается как расширенная роль.
- Трекер/Ментор/Команда — расширенные роли для проектов (детализация на следующих этапах).

Подробная матрица прав будет проработана в будущей редакции.

## 3. Функциональные блоки

### 3.1. База запросов на инновации
- Создание запроса стейкхолдером: название, проблема/цель, критерии принятия решения, сроки сбора идей, владелец, вложения, направление (1. Строительство; 2. Эксплуатация; 3. Безопасность; 4. Цифровизация; 5. Клиентский сервис; 6. «Свободный микрофон»).
- Статусы: Черновик → На сборе идей → Закрыт.
- Каталог запросов: карточки, фильтры/поиск, сортировки, страница запроса.
- Связь с идеями (см. 3.2). Уведомления участникам о старте/закрытии.

### 3.2. База идей (привязанных к запросам)
- Подача идеи участником на конкретный запрос (типично: 1 абзац + опциональные вложения/ссылки).
- Поля: заголовок, краткое описание (≤ 1000 знаков), запрос‑связка, автор, дата, теги.
- Статусы: Новая → На уточнении (модерация) → Принята стейкхолдером → Отклонена (на доработку/в корзину).
- Комментарии/обсуждения (участники, стейкхолдеры, модераторы; эксперты‑эдвайзеры).

### 3.3. База проектов
- Преобразование идеи в проект после подтверждения стейкхолдером.
- Поля проекта: цель/результат, владелец‑стейкхолдер, трекер, состав команды, сроки/этапы, связи (исходный запрос и идея), статус.
- Рабочая документация: паспорт проекта, план работ, протоколы, риск‑лог, артефакты исследований/POC, презентация на защиту.
- Регламент стадий: Инициация → Планирование → Исполнение → Подготовка к защите/закрытие.
- Файлы и версионирование на средствах Bitrix (Диск/Документы).

### 3.4. База участников процесса
- Справочник: стейкхолдеры, менторы, трекеры, эксперты, команды.
- Профили (минимум): ФИО, роль(и), контакты, отдел/компания, навыки/интересы (теги), участие в проектах.
- Поиск/фильтры; карточка участника/команды.

### 3.5. Сервис совместной работы (Bitrix)
- Совместное редактирование документов, хранение файлов по структуре проекта.
- Трекинг задач: шаблоны задач, чек‑листы, канбан, исполнители/наблюдатели, сроки/напоминания.
- Комментарии, @упоминания, уведомления, история изменений.
- Базовая отчетность: задачи по статусам, активность команды.

### 3.6. Сервис «Доска объявлений / Расписание»
- Типы мероприятий: конкурсы/хакатоны, ярмарки вакансий, семинары, совещания, конференции и др.
- Каталог + календарь, карточка события, регистрация/ссылки, напоминания.

### 3.7. Адаптация интерфейсов под дизайн‑макеты Заказчика
- Применение предоставленного гайда/макетов: типографика, цвета, сетка, компоненты.
- Адаптивная вёрстка ключевых страниц.

## 4. Этапы и объем работ

### Фаза 1 — MVP «Запросы и Идеи»
- Цель: запустить базовый цикл «стейкхолдер формирует запрос → участники подают идеи».
- База запросов: создание/публикация, каталог, фильтры, статусы.
- База идей: подача на конкретный запрос, модерация, статусы, комментарии.
- Роли Ф1: стейкхолдер, участник, модератор.
- Уведомления (e‑mail/внутренние) по ключевым событиям.
- Минимальная навигация/главная страница платформы.
- Публичная витрина платформы.

### Фаза 2 — Проекты и Совместная работа
- Цель: обеспечить трансформацию подтвержденной идеи в проект и базовую проектную работу в Bitrix.
- Превращение идеи в проект, карточка проекта.
- Справочник участников: базовые профили, команды; привязка к проектам.
- Шаблоны проектных документов (паспорт, план, протокол, риск‑лог, шаблон презентации).
- Роли Ф2: трекер, ментор, эксперт.
- Личные кабинеты ролей.
- Интеграция с задачами Bitrix: доски, чек‑листы, напоминания, базовая отчётность.
- Навигация по проектам, статусы стадий проекта.

### Фаза 3 — События, аналитика, уведомления и полная адаптация дизайна
- Цель: коммуникации и планирование активности, доведение UX/UI и бренд‑адаптации.
- «Доска объявлений/Расписание»: каталог, календарь, карточка, регистрация/ссылки, напоминания.
- Расширение профилей (навыки/интересы, участие в проектах), поиск/фильтры.
- Статистика и аналитика: публичные метрики, личная статистика, админ‑панель (воронка, активность, экспорт отчетов).
- Система уведомлений: email/внутренние уведомления, счетчик, история, настройки подписок, еженедельный дайджест.
- Полная адаптация под дизайн‑макеты Заказчика, полировка UX/UI.
- Архивация завершенных проектов.

## 5. Нефункциональные требования
- Платформа: Bitrix (Сайты/Диск/Задачи/Группы) — преимущественно стандартные средства, кастомизация по минимуму.
- Производительность: ключевые страницы ≤ 2 с при средней нагрузке MVP.
- Безопасность: разграничение доступа по ролям; передача доступов по защищенным каналам; аудит действий в рамках Bitrix.
- Совместимость: последние 2 мажорные версии браузеров.

## 6. Интеграции и ограничения
- Внешние защиты на НТС, финансовые блоки, сложная аналитика — за пределами текущего объема.
- Почтовые уведомления — стандартные средства Bitrix.

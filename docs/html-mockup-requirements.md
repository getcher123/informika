# Требования к HTML/CSS/JS макетам для кастомных Bitrix-страниц

Документ предназначен для внешнего исполнителя, который нарезает макеты в чистый HTML/CSS/JS. Эти макеты затем будут адаптированы под шаблоны Bitrix. Цель — получить код, максимально близкий к будущим PHP-шаблонам.

## 1. Общие принципы
- Семантичный HTML5, строгий DOCTYPE, корректная кодировка UTF-8.
- Вся разметка ведётся BEM или согласованным неймингом (блок__элемент--модификатор).
- Структурные элементы (header, main, footer, section, article, nav, aside) используем по назначению.
- Любые интерактивные элементы — через `<button>` или `<a>` с правильными атрибутами (`role`, `aria-*` при необходимости).
- Не использовать inline-стили и inline-скрипты — всё в отдельных файлах.

## 2. Организация файлов

```
/project-mockup/
  index.html
  requests.html
  request-detail.html
  idea-detail.html
  team-requests.html
  stake-holder-dashboard.html
  assets/
    css/
      base.css
      layout.css
      components.css
      pages/
        requests.css
        request-detail.css
        idea-detail.css
        …
    js/
      vendor.js
      app.js
      requests.js
      idea.js
    fonts/
    img/
```

- Каждая страница — отдельный HTML-файл, имя отражает содержание.
- Общие стили (reset, variables, typography) хранятся в `base.css`, сетки/контейнеры — в `layout.css`, UI-кит — в `components.css`.
- Для страницы — отдельный файл в `assets/css/pages/`.
- JS — модульный: `app.js` (инициализация), `requests.js`, `idea.js` и т.д. Внешние библиотеки — в `vendor.js` (с указанием версии и источника).
- Изображения оптимизированы (webp/avif, fallback png/jpg), шрифты — woff2 + woff.

## 3. HTML-требования
- Поддержка адаптивности: mobile-first, брейкпоинты на 320/768/1024/1440.
- Использовать `picture` для responsive images (при необходимости).
- Формы (`Подать идею`, `Вступить в команду`, `Отправить на доработку`) должны содержать:
  - `<label>` + `for` для всех полей.
  - Placeholder/description (если в макете), required-атрибуты, подсказки, блок ошибок.
- Модальные окна реализовать как отдельные `div` с `role="dialog"`, скрываются/показываются JS (без inline-логики).
- Таблицы, списки, карточки — по семантике (`table`, `ul/li`, `article`).
- Для индикаторов («• Новые», «+N», бейджи стадий) подготовить соответствующие элементы (`span` с классами, aria-label).
- Навигация: хлебные крошки (если в макете) — список (`nav > ol`).
- Ссылки `href` должны быть заполнены «#» (не пустыми) и иметь `data-target` или класс для JS привязки.



## 4. CSS-требования
- Препроцессор (например, SCSS) приветствуется, но финальные файлы — чистый CSS.
- Обязательно использование CSS variables для цветов, отступов, шрифтов (упростит темизацию).
- Использовать flex/grid для макетирования.
- Стили должны быть изолированы — никаких глобальных правил по элементам (`div { … }`).
- Переменные и mixins (если SCSS) — в `base`; цветовые модификаторы (бейджи) описать в отдельном блоке.
- Анимации и переходы — по необходимости, но лёгкие (transition < 0.2s).
- Правила для состояний (`:hover`, `:focus`, `:disabled`) обязательны.
- Не использовать CSS-фреймворки (Bootstrap и т.п.), если не согласовано отдельно.

## 5. JS-требования
- Писать на ES6+; если нужны полифилы — описать отдельно.
- Модульная структура: инициализация на DOMContentLoaded, лёгкий router (если страниц много).
- Использовать vanilla JS или небольшие библиотеки (например, Alpine.js, Swiper). Не тянуть тяжёлый фреймворк.
- Модальные окна, выпадающие списки, индикаторы — управляются через классы (`is-open`, `is-active`).
- Формы: работа состояния (disabled кнопок, подсветка ошибок) должна быть показана. Реальную отправку можно имитировать (data attributes и комментарий).
- Логику окружности (например, счётчики «+N») реализовать как можно ближе к дизайн-индикаторам (в JS описать класс, который умеет менять числа, подсветку).
- JS должен работать без внешних сборщиков при подключении `vendor.js` и `app.js`.

## 6. Адаптивность и доступность
- Breakpoints минимум: 320px, 768px, 1024px, 1440px.
- Использовать `rem`/`em` для типографики и отступов.
- Проверить доступность:
  - Контраст кнопок/текстов.
  - Навигация по tab.
  - aria-label для иконок (например, индикаторы, кнопка «Закрыть»).
- Развернуть бургер-меню, если в макете есть мобильное меню; модальные окна должны полноценно работать на touch.

## 7. Проверка и поставка
- Предоставить:
  - Архив с HTML/CSS/JS (см. структуру).
  - Документацию: список страниц, используемых библиотек, версии.
  - Скриншоты ключевых breakpoint’ов (desktop/tablet/mobile).
  - Описание, как инициализируются модальные окна, выпадающие списки, индикаторы.
- Все пути в HTML относительные (`./assets/...`).
- Не использовать SSG/SSR — всё статические файлы.
- Зависимости npm (если нужны) — перечислить в `package.json` (разрешено, но финал должен собираться в чистый html/css/js).
